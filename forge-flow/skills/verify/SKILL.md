구현 완료된 코드를 design 문서 기준으로 **코드 수준 검증**합니다.
규모에 따라 검수 방식이 달라집니다.

> 실제 런타임 테스트(브라우저 UI/UX, API 호출)는 `/forge-flow:test`에서 수행합니다.

## 선행 조건 검사

실행 전 반드시 확인:
1. 상태 파일 존재 → 없으면: "워크플로가 시작되지 않았습니다. `/forge-flow:clarify`로 시작하세요."
2. phase가 `"implementing"` 또는 `"verifying"`인지 확인 → 아니면: "현재 `{phase}` 단계입니다. 구현 완료 후 verify를 실행하세요."
3. `design_file`이 존재하는 파일 경로인지 확인 → 없으면: "설계 문서를 찾을 수 없습니다."

## 상태 파일 갱신

실행 시작 시:
```json
{ "phase": "verifying", "rework_count": 0 }
```

> `rework_count`는 verify 진입 시 0으로 초기화합니다 (이전 phase의 rework과 독립).

## 규모별 검수 방식

### S 규모: 경량 검수

1. **빌드 검증**: `/forge-flow:build-check` 실행. CLAUDE.md에 `## FE 빌드 명령` 섹션이 있으면 `/forge-flow:fe-check`도 실행.
2. **AC 대조**: design 문서의 AC 항목을 하나씩 코드와 대조
3. 교차검증 생략

### M 규모: 표준 검수

#### 1단계: 메인 세션 검수

1. **빌드 검증**: `/forge-flow:build-check` 실행. CLAUDE.md에 `## FE 빌드 명령` 섹션이 있으면 `/forge-flow:fe-check`도 실행.
2. **코드 리뷰** (diff 기반):
   - 변경된 코드가 design 문서의 구현 계획과 일치하는가?
   - **기존 코드 패턴 일관성 검증** (design 문서의 `따를 기존 패턴` 섹션 기준):
     - 네이밍 컨벤션이 기존과 동일한가?
     - 에러 처리 방식이 기존 패턴을 따르는가?
     - 파일/폴더 구조가 기존 구조와 일관되는가?
     - import 순서, 로깅, 테스트 작성 방식이 일치하는가?
   - 기존 패턴과 다르게 작성된 코드 발견 시 → **REWORK** (사용자가 패턴 변경을 요청한 경우 제외)
3. **AC 대조**: 각 AC 항목을 코드에서 실제로 처리하는 위치 확인
4. **사이드이펙트 검토**: 변경이 다른 모듈에 미치는 영향
5. **엣지케이스 리뷰**: 경계값, null/empty, 실패 경로 처리

#### 2단계: 교차검수 (에이전트팀 또는 서브에이전트)

에이전트팀 활성화 여부에 따라 분기합니다.

---

**A. 에이전트팀 활성화 시: 동적 검증 팀 구성**

> **에이전트팀 기능**으로 검증 팀을 구성합니다. Agent tool 서브에이전트가 아닌, 에이전트팀의 독립 프로세스로 각 검증자를 spawn합니다.

검증 관점을 분석하여 팀원을 동적으로 구성합니다:

**검증 팀 구성 분석**:

| 분리 기준 | 예시 |
|----------|------|
| **검증 관점 분리** | AC 충족 검증 / 패턴·사이드이펙트 검증 |
| **도메인 분리** | 사용자단 시나리오 / 관리자단 시나리오 |
| **레이어 분리** | BE 검증 (API·로직) / FE 검증 (UI·인터랙션) |
| **코드 관점 분리** | 로직 검증 / 에러 처리 검증 / 타입 안전성 검증 |

독립 검증 관점이 2개 이상이면 → 검증 팀 구성. 1개면 → 서브에이전트 폴백(B).

**검증 팀 역할 정의**:

| 역할 | 책임 |
|------|------|
| **검증자-AC** | AC 항목별 코드 위치(파일:라인) 1:1 매핑, 구현 누락 검출 |
| **검증자-패턴** | design 문서의 `따를 기존 패턴` 기준 일관성 검증, 사이드이펙트, 엣지케이스 검증 |

> 역할은 프로젝트와 작업에 따라 **동적으로 결정**합니다. 위는 예시이며, 실제로는 작업 특성에 맞게 조합합니다.
> 실제 동작 테스트(UI/UX, API)는 verify가 아닌 `/forge-flow:test`에서 수행합니다.

**검증 팀 구성 보고**:
```
[검증 팀 구성]
  검증자 A: AC 충족 검증 — 각 AC를 코드와 1:1 대조
  검증자 B: 패턴·사이드이펙트 — 코드 품질 + 영향 범위
  검증자 C: 사용자단/관리자단 시나리오 검증

검증 팀을 구성합니다.
```

**검증자 프롬프트 템플릿** (model 생략 → 세션 모델 상속):
```
당신은 {프로젝트명}의 독립 검증자입니다.
검증 관점: {담당 관점}

## design 문서
{design 문서 전문}

## 변경 내용
{git diff 결과}

## 검증 항목
{담당 관점에 해당하는 구체적 검증 항목}

## 응답 형식
각 항목에 대해:
- PASS: <근거 한 줄>
- FAIL: <구체적 문제점 + 수정 제안>

## 종합 판정
PASS / CONCERNS / REWORK / FAIL + 근거
```

> **핵심 원칙**: 메인 세션의 검수 결과를 검증 팀원에게 전달하지 않음 (독립 판단 보장).

**검증 팀 결과 종합** (리더):
1. 모든 검증자의 태스크 완료 대기
2. 각 검증자의 판정 결과 수집
3. 메인 세션의 1단계 검수 결과와 교차 대조
4. 최종 판정 도출

---

**B. 에이전트팀 비활성화 시: 서브에이전트 교차검수 (폴백)**

서브에이전트로 검수자를 spawn합니다 (Agent tool, `isolation: "worktree"`, model 생략 → 세션 모델 상속):

```
당신은 독립 검수자입니다. 아래 design 문서와 코드 변경(diff)을 검토하고, 각 항목에 PASS/FAIL + 근거를 반환하세요.

## 검토 항목

1. **AC 충족**: 각 AC가 코드에서 구현되었는가?
   - AC 항목별로 해당 코드 위치(파일:라인) 명시
   - 구현 누락 시 FAIL

2. **패턴 일관성**: 기존 프로젝트 코드와 패턴이 일치하는가?
   - design 문서의 `따를 기존 패턴` 섹션에 명시된 패턴과 실제 구현 대조
   - 네이밍 컨벤션, 에러 처리, 파일 구조, import 순서, 로깅, 테스트 방식
   - 기존 패턴과 다르게 작성된 코드 발견 시 FAIL 판정 (사용자의 패턴 변경 요청이 없는 한)

3. **사이드이펙트**: 의도하지 않은 동작 변경이 있는가?
   - 공유 상태(DB, API, 이벤트) 영향 확인

4. **엣지케이스**: 경계값, null/empty, 실패 경로, 동시성 고려

5. **누락**: design 문서의 구현 계획에는 있지만 실제 변경에 없는 파일이 있는가?

## 응답 형식

각 항목에 대해:
- PASS: <근거 한 줄>
- FAIL: <구체적 문제점 + 수정 제안>

## 종합 판정
PASS / CONCERNS / REWORK / FAIL + 근거

[design 문서]
{design 문서 전문}

[변경 내용]
{git diff 결과}
```

> **핵심 원칙**: 메인 세션의 검수 결과를 서브에이전트에게 전달하지 않음 (독립 판단 보장).

---

#### 3단계: 결과 종합

**에이전트팀 사용 시 (다중 검증)**:

| 검수 주체 | 관점 | 특성 |
|----------|------|------|
| **verify (메인 세션)** | 정량적 | 빌드 + 체크리스트, "AC 5개 중 5개?" |
| **검증 팀원 A** | 독립 관점 A | design + diff 기반, 편향 없음 |
| **검증 팀원 B** | 독립 관점 B | design + diff 기반, 편향 없음 |
| **리뷰어 (구현 팀)** | 정성적 | 구현 중 상시 리뷰 결과 (있는 경우) |

> 구현 단계에서 에이전트팀을 사용한 경우, 리뷰어의 리뷰 결과도 함께 반영합니다.

**최종 판정**:
- 전원 통과 → 작업 완료
- 일부 실패 → 실패 항목 정리 → 수정 후 재검수
- 요구사항/설계 자체 문제 → 사용자에게 재검토 요청 → clarify부터 재진행
- 에이전트팀 세션 종료 → 서브에이전트 폴백(B)으로 재검수

**에이전트팀 비활성화 시 (2중 검증)**:
- 메인 + 서브에이전트 양쪽 통과 → 작업 완료
- 한쪽 불합격 → 불합격 항목 기반 수정 후 재검수
- 요구사항/설계 자체 문제 → 사용자에게 재검토 요청

### L 규모: 전체 검수

M 규모의 모든 검수 항목을 포함하며, 추가로:

1. **아키텍처 영향 검토**: 모듈 간 의존성 변경, API 계약 변경이 설계대로 반영되었는가?
2. **변경 전파 체인 검증**: CLAUDE.md의 `## 변경 전파 체인` 테이블 기반으로, 전파 대상이 모두 변경되었는가?
3. **AC 전수 검사**: 모든 AC를 코드 위치(파일:라인)와 1:1 대응시켜 검증

> L 규모에서는 에이전트팀 활성화 여부와 관계없이 **검증 팀 구성을 권장**합니다. 에이전트팀 비활성화 시에도 서브에이전트를 복수 spawn하여 관점별 교차검증을 수행합니다.

## 품질 게이트

| 등급 | 기준 | 조치 |
|------|------|------|
| **통과**(PASS) | 전 항목 통과 | 작업 완료 |
| **주의**(CONCERNS) | 경미한 이슈 (코딩 스타일 등) | `AskUserQuestion`으로 판단 위임 |
| **재작업**(REWORK) | AC 미충족, 패턴 불일치 등 | 해당 부분 수정 → 재검수 |
| **실패**(FAIL) | 근본적 문제 | 이전 단계부터 재검토 |

> **재작업 연속 3회 시 실패로 에스컬레이션.**

**주의 판정 시 AskUserQuestion 호출**:
```
question: "경미한 이슈가 발견되었습니다. 어떻게 진행할까요?"
header: "주의"
options:
  - label: "수용 — 진행 (Recommended)"
    description: "경미한 이슈를 인지하고 다음 단계로 진행합니다"
  - label: "수정 후 재검수"
    description: "이슈를 수정한 뒤 verify를 다시 실행합니다"
multiSelect: false
```

**재작업 처리 흐름**:
1. 문제점을 사용자에게 보고 (수정 필요한 파일과 내용 명시)
2. 상태 파일의 `rework_count`를 +1 (`rework_count` ≥ 3이면 실패로 에스컬레이션)
3. phase를 `"implementing"`으로 되돌림
4. 사용자가 코드 수정 (또는 Claude가 수정 제안)
5. 수정 완료 후 사용자가 `/forge-flow:verify` 재호출

## 검수 결과 기록

design 문서의 `## 검수 결과`, `## 검수 이력` 섹션에 기록:

```markdown
## 검수 결과
- verify: PASS (2026-03-13)

## 검수 이력
### verify
- #1 REWORK: 기존 ErrorHandler 패턴과 불일치
  → 수용: try-catch → ErrorHandler.wrap() 패턴으로 변경
- #2 PASS
```

## 상태 파일 갱신 (완료 시)

PASS 시:
```json
{ "phase": "verified", "stop_count": 0, "rework_count": 0 }
```

REWORK 시:
```json
{ "phase": "implementing", "rework_count": +1 }
```

FAIL 시:
```json
{ "phase": "clarifying" }
```
> REWORK/FAIL로 인한 후퇴 시 `stop_count`를 리셋하지 않습니다.
> FAIL 시 `rework_count`는 clarify 재진입 시 0으로 초기화됩니다.

## 완료 후 다음 단계

verify PASS 후 → 사용자의 추가 입력 없이 **즉시 `/forge-flow:test`를 호출**합니다.

> `/forge-flow:test`에서 스킵 조건에 해당하면 자동으로 스킵되므로, verify는 항상 test를 호출합니다.

> 완료된 작업의 스펙과 검수 이력은 커밋 메시지와 코드에 반영되어 있으므로 별도 보존 불필요.
