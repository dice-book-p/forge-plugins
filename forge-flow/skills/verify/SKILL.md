구현 완료된 코드를 design 문서 기준으로 **코드 수준 검증**합니다.
규모에 따라 검수 방식이 달라집니다.

> 실제 런타임 테스트(브라우저 UI/UX, API 호출)는 `/forge-flow:test`에서 수행합니다.

## 선행 조건 검사

실행 전 반드시 확인:
1. 현재 세션에 바인딩된 상태 파일 탐색 → `.forge-flow/state/`에서 `session_id`가 현재 세션(`${CLAUDE_SESSION_ID}`)과 일치하는 `{task_id}.json` 파일 탐색 → 없으면: "워크플로가 시작되지 않았습니다. `/forge-flow:clarify`로 시작하세요."
2. phase가 `"implementing"` 또는 `"verifying"`인지 확인 → 아니면: "현재 `{phase}` 단계입니다. 구현 완료 후 verify를 실행하세요."
3. `design_file`이 존재하는 파일 경로인지 확인 → 없으면: "설계 문서를 찾을 수 없습니다."

## 상태 파일 갱신

실행 시작 시:
```json
{ "phase": "verifying" }
```

> `rework_count`는 **최초 verify 진입 시에만** 0으로 초기화합니다. REWORK 후 재진입 시에는 기존 `rework_count`를 유지합니다 (3회 에스컬레이션 추적을 위해).
> 최초 진입 판단: 이전 phase가 `"implementing"`이면 최초 진입 → `rework_count: 0`. 이전 phase가 `"verifying"`이면 재진입 → 유지.

## 검증 강도 확인

design 문서의 `## 검증 강도` 섹션을 확인합니다. 검증 강도는 교차검수(2단계) 수준을 결정합니다:

- **표준 (1)**: 에이전트팀 1명으로 교차검수.
- **강화 (2+)**: 에이전트팀 관점별 복수 검증자로 교차검수.

> 검증 강도가 미설정이면 규모 기반 기본값을 적용합니다 (S=1, M=1, L=2).
> 모든 규모에서 최소 강도 1 — 외부 교차검수 필수.
> 메인 세션 검수(1단계)의 깊이는 규모에 따라 결정됩니다 (S/M/L).

## 규모별 검수 방식

### S 규모: 경량 검수

1. **빌드 검증**: `/forge-flow:build-check` 실행. CLAUDE.md에 `## FE 빌드 명령` 섹션이 있으면 `/forge-flow:fe-check`도 실행.
2. **AC 대조**: design 문서의 AC 항목을 하나씩 코드와 대조
3. **에이전트팀 1명 교차검수**: 에이전트팀 검증자 1명을 spawn하여 AC 충족 + 패턴 일관성을 독립 검증 (M 규모 2단계의 축소판)
4. **결과 종합**: 메인 세션 + 검증자 결과를 대조하여 최종 판정

### M 규모: 표준 검수

#### 0단계: rework-log 참조

`.forge-flow/rework-log.md` 존재 시 스캔하여 이번 작업 영향 범위와 관련된 ×2+ 패턴 확인. 있으면 검증자 프롬프트에 포함.

#### 1단계: 메인 세션 검수

1. **빌드 검증**: `/forge-flow:build-check` 실행. CLAUDE.md에 `## FE 빌드 명령` 섹션이 있으면 `/forge-flow:fe-check`도 실행.
2. **코드 리뷰** (diff 기반):
   - 변경된 코드가 design 문서의 구현 계획과 일치하는가?
   - **기존 코드 패턴 일관성 검증** (design 문서의 `따를 기존 패턴` 섹션 기준):
     - 네이밍 컨벤션이 기존과 동일한가?
     - 에러 처리 방식이 기존 패턴을 따르는가?
     - 파일/폴더 구조가 기존 구조와 일관되는가?
     - import 순서, 로깅, 테스트 작성 방식이 일치하는가?
   - 기존 패턴과 다르게 작성된 코드 발견 시 → **REWORK** (사용자가 패턴 변경을 요청한 경우 제외)
3. **AC 대조**: 각 AC 항목을 코드에서 실제로 처리하는 위치 확인
4. **사이드이펙트 검토**: 변경이 다른 모듈에 미치는 영향
5. **엣지케이스 리뷰**: 경계값, null/empty, 실패 경로 처리

#### 2단계: 에이전트팀 교차검수

검증 강도에 따라 팀 규모를 결정합니다:

```
강도 1 → 에이전트팀 1명
강도 2+ → 에이전트팀 (관점별 복수)
```

**에이전트팀 교차검수**

에이전트팀으로 검증 팀을 구성합니다. 각 검증자는 독립 Claude Code 프로세스로 실행됩니다.

**에이전트팀 spawn 절차**:

1. **`TeamCreate` 호출**: 팀 생성
   ```json
   { "team_name": "verify-{task_id}", "description": "코드 검증 팀" }
   ```

2. **`TaskCreate`로 검증 작업 생성**: 각 검증 관점별로 작업 생성

3. **`Agent` 도구로 팀원 spawn**: 반드시 `team_name`과 `name` 파라미터를 지정하여 에이전트팀 팀원으로 등록
   ```
   Agent(
     team_name: "verify-{task_id}",
     name: "검증자-AC",
     prompt: "...(아래 템플릿)",
     model: "sonnet"
   )
   ```
   > `team_name` + `name`으로 에이전트팀 팀원을 등록합니다.

4. **결과 수신**: 팀원의 메시지가 자동 전달됨 (폴링 불필요)

5. **`SendMessage`로 종료 요청** → **`TeamDelete`로 정리**

**검증 팀 구성 분석**:

| 분리 기준 | 예시 |
|----------|------|
| **검증 관점 분리** | AC 충족 검증 / 패턴·사이드이펙트 검증 |
| **도메인 분리** | 사용자단 시나리오 / 관리자단 시나리오 |
| **레이어 분리** | BE 검증 (API·로직) / FE 검증 (UI·인터랙션) |
| **코드 관점 분리** | 로직 검증 / 에러 처리 검증 / 타입 안전성 검증 |

독립 검증 관점이 2개 이상이면 → 복수 팀원 구성. 1개면 → 팀원 1명으로 구성.

**검증 팀 역할 정의**:

| 역할 | 책임 |
|------|------|
| **검증자-AC** | AC 항목별 코드 위치(파일:라인) 1:1 매핑, 구현 누락 검출 |
| **검증자-패턴** | design 문서의 `따를 기존 패턴` 기준 일관성 검증, 사이드이펙트, 엣지케이스 검증 |

> 역할은 프로젝트와 작업에 따라 **동적으로 결정**합니다. 위는 예시이며, 실제로는 작업 특성에 맞게 조합합니다.
> 실제 동작 테스트(UI/UX, API)는 verify가 아닌 `/forge-flow:test`에서 수행합니다.

**검증 팀 구성 보고**:
```
[검증 팀 구성 — 에이전트팀]
  team: verify-{task_id}
  검증자 A: AC 충족 검증 — 각 AC를 코드와 1:1 대조
  검증자 B: 패턴·사이드이펙트 — 코드 품질 + 영향 범위

검증 팀을 spawn합니다.
```

**검증자 프롬프트 템플릿** (`model: "sonnet"`):
```
당신은 {프로젝트명}의 독립 검증자입니다.
검증 관점: {담당 관점}

## design 문서
{design 문서 전문}

## 따를 기존 패턴
{design 문서의 '따를 기존 패턴' 테이블}

## 변경 내용
{git diff 결과}

## 검증 항목
{담당 관점에 해당하는 구체적 검증 항목}

## 검증 방법 (해당 시)
{design 문서의 '검증 방법' 테이블에서 담당 항목}

## 이 프로젝트에서 반복되는 실수 패턴 (해당 시)
{rework-log에서 이번 작업 영향 범위와 관련된 ×2+ 항목 발췌}

위 패턴에 특히 주의하여 검증하세요.

## 응답 원칙
- PASS 항목은 한 줄로 요약 (상세 불필요)
- FAIL 항목만 상세히: 문제 + 위치(file:line) + 수정 제안
- 전체 응답 2,000 토큰 이내 권장

## 응답 형식
각 항목에 대해:
- PASS: <근거 한 줄>
- FAIL: <구체적 문제점 + 위치(file:line) + 수정 제안>

## 종합 판정
PASS / CONCERNS / REWORK / FAIL + 근거
```

> **핵심 원칙**: 메인 세션의 검수 결과를 검증 팀원에게 전달하지 않음 (독립 판단 보장).

**검증 팀 결과 종합 — 숙의(Deliberation)** (리더):
1. 모든 검증자의 메시지 자동 수신 대기 (팀원 idle 알림으로 완료 감지)
2. 각 검증자의 판정 결과 수집
3. **불일치 식별**: 메인 세션 vs 검증자 간, 또는 검증자 간 판정이 다른 항목 목록화
4. **갈등 분석** (불일치 항목이 있을 때):
   - 각 불일치에 대해 "어느 관점이 더 타당한가?" 추론
   - 근거: 코드 증거(파일:라인), AC 원문, 패턴 기준 중 어느 것에 기반하는지 명시
5. **판정 근거 작성**: 최종 판정에 대해 "왜 이 결정을 내렸는지" 기록
6. 최종 판정 도출

**숙의 결과 보고 형식**:
```
[숙의 결과]
  일치 항목: AC-1(PASS), AC-3(PASS), ...
  불일치 항목:
    - AC-2: 메인(PASS) vs 검증자-패턴(FAIL) → 채택: FAIL
      근거: ErrorHandler.wrap() 패턴 미사용 (src/handler.ts:45)
  판정 근거: AC-2 패턴 불일치로 REWORK 판정
```

#### 3단계: 결과 종합

| 검수 주체 | 관점 | 특성 |
|----------|------|------|
| **verify (메인 세션)** | 정량적 | 빌드 + 체크리스트, "AC 5개 중 5개?" |
| **검증 팀원 A** | 독립 관점 A | design + diff 기반, 편향 없음 |
| **검증 팀원 B** | 독립 관점 B | design + diff 기반, 편향 없음 |
| **리뷰어 (구현 팀)** | 정성적 | 구현 중 상시 리뷰 결과 (있는 경우) |

> 구현 단계에서 에이전트팀을 사용한 경우, 리뷰어의 리뷰 결과도 함께 반영합니다.

**최종 판정**:
- 전원 통과 → 작업 완료
- 일부 실패 → 실패 항목 정리 → 수정 후 재검수
- 요구사항/설계 자체 문제 → 사용자에게 재검토 요청 → clarify부터 재진행

### L 규모: 전체 검수

M 규모의 모든 검수 항목을 포함하며, 추가로:

1. **아키텍처 영향 검토**: 모듈 간 의존성 변경, API 계약 변경이 설계대로 반영되었는가?
2. **변경 전파 체인 검증**: CLAUDE.md의 `## 변경 전파 체인` 테이블 기반으로, 전파 대상이 모두 변경되었는가?
3. **AC 전수 검사**: 모든 AC를 코드 위치(파일:라인)와 1:1 대응시켜 검증

> L 규모에서는 에이전트팀으로 **관점별 복수 검증자 구성이 필수**입니다.

## 품질 게이트

| 등급 | 기준 | 조치 |
|------|------|------|
| **통과**(PASS) | 전 항목 통과 | 작업 완료 |
| **주의**(CONCERNS) | 경미한 이슈 (코딩 스타일 등) | `AskUserQuestion`으로 판단 위임 |
| **재작업**(REWORK) | AC 미충족, 패턴 불일치 등 | 해당 부분 수정 → 재검수 |
| **실패**(FAIL) | 근본적 문제 | 이전 단계부터 재검토 |

> **재작업 연속 3회 시 실패로 에스컬레이션.**

**주의 판정 시 AskUserQuestion 호출**:
```
question: "경미한 이슈가 발견되었습니다. 어떻게 진행할까요?"
header: "주의"
options:
  - label: "수용 — 진행 (Recommended)"
    description: "경미한 이슈를 인지하고 다음 단계로 진행합니다"
  - label: "수정 후 재검수"
    description: "이슈를 수정한 뒤 verify를 다시 실행합니다"
multiSelect: false
```

**재작업 처리 흐름**:
1. 문제점을 사용자에게 보고 (수정 필요한 파일과 내용 명시)
2. **rework-log 기록** (아래 규칙 참조)
3. 상태 파일의 `rework_count`를 +1 (`rework_count` ≥ 3이면 실패로 에스컬레이션)
4. phase를 `"implementing"`으로 되돌림
5. 사용자가 코드 수정 (또는 Claude가 수정 제안)
6. 수정 완료 후 사용자가 `/forge-flow:verify` 재호출

## Rework Log 관리

REWORK 판정 시 `.forge-flow/rework-log.md`에 패턴을 기록합니다. clarify/plan에서 이 로그를 참조하여 동일 실수를 예방합니다.

**기록 절차**:
1. `.forge-flow/rework-log.md` 읽기 (없으면 생성)
2. 기존 항목과 **유사 패턴** 검색 (같은 원인 유형)
3. 유사 패턴 있으면 → 카운트 +1, 날짜 갱신
4. 새 패턴이면 → 항목 추가
5. **정리**: 추가 후 아래 관리 규칙 적용

**항목 형식**:
```markdown
## {원인 요약} (×{횟수})
- 최근: {날짜} | {verify/test}
- 파일: {관련 파일}
- 교훈: {재발 방지 핵심 한 줄}
<!-- first: {최초 발생일} -->
```

**관리 규칙**:
- 최대 **10건** 유지 — 초과 시 카운트(×N)가 가장 낮은 항목부터 삭제
- 단발성(×1) 항목은 **30일** 경과 시 삭제
- 반복 패턴(×2+)은 TTL 무관 유지

## 검수 결과 기록

design 문서의 `## 검수 결과`, `## 검수 이력` 섹션에 기록:

```markdown
## 검수 결과
- verify: PASS (2026-03-13)

## 검수 이력
### verify
- #1 REWORK: 기존 ErrorHandler 패턴과 불일치
  → 수용: try-catch → ErrorHandler.wrap() 패턴으로 변경
- #2 PASS
```

## 상태 파일 갱신 (완료 시)

PASS 시:
```json
{ "phase": "verified", "stop_count": 0, "rework_count": 0 }
```

REWORK 시:
```json
{ "phase": "implementing", "rework_count": +1 }
```

FAIL 시:
```json
{ "phase": "clarifying" }
```
> REWORK/FAIL로 인한 후퇴 시 `stop_count`를 리셋하지 않습니다.
> FAIL 시 `rework_count`는 clarify 재진입 시 0으로 초기화됩니다.

## 완료 후 다음 단계
