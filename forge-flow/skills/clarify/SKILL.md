구현 착수 전 요구사항을 명확한 스펙으로 변환하고 예비 규모를 판정합니다.

## 프로젝트 업데이트 확인

CLAUDE.md의 `<!-- forge-flow:version=X.X.X -->` 마커를 확인합니다.
마커 버전이 **`3.4.7`** 미만이면 아래 안내를 표시한 뒤 **정상 진행**합니다 (차단하지 않음):

```
⚠️ forge-flow 프로젝트 설정 업데이트가 필요합니다.
   현재: v{마커 버전} → 최신: v{이 값 이상}
   `/forge-flow:setup-workflow --update` 를 실행하세요.
   (지금 작업은 정상 진행됩니다)
```

> 마커가 없거나 버전이 같거나 높으면 이 섹션을 건너뜁니다.

## 필수 게이트

**스펙 없이 구현된 코드는 방향이 잘못될 수 있습니다. 재작업 비용 > 질문 비용.**

`/forge-flow:clarify`는 선택이 아닌 **필수 진입 게이트**입니다.
모든 작업 시작 전, 예외 없이 첫 번째 단계로 실행합니다.

| 시점 | 이유 |
|------|------|
| **모든 작업 시작 전** | 요구사항이 확정되지 않으면 구현 방향 자체가 틀릴 수 있음 |
| **작업 중 요구사항이 바뀐 순간** | 기존 스펙 무효화 → 즉시 중단 → 재실행 |
| **verify 실패 원인이 요구사항 불명확** | verify에서 재실행 요청됨 |

## 작업 탐색 (Task Discovery)

실행 시 `.forge-flow/state/` 디렉토리를 스캔하여 기존 작업을 확인합니다.
> `.forge-flow/` 디렉토리가 없으면 생성하고 스캔을 건너뜁니다.

### 시나리오 분기

**A. 현재 세션에 바인딩된 작업 존재** (`session_id` 일치)

**AskUserQuestion 호출**:
```
question: "이전 작업 '{task_id}'이(가) '{phase}' 단계에 있습니다. 어떻게 진행할까요?"
header: "기존 작업 발견"
options:
  - label: "이어서 진행 (Recommended)"
    description: "이전 작업을 현재 단계부터 이어서 진행합니다"
  - label: "추가 요구사항 반영"
    description: "이전 design에 요구사항을 추가/수정한 뒤 재검수합니다"
  - label: "새 작업 시작"
    description: "이전 작업을 보관하고 새 작업을 시작합니다"
multiSelect: false
```

- "이어서 진행" → design 파일을 읽고 해당 phase의 다음 단계를 안내합니다:
  - implementing → "구현을 이어서 진행하세요. 완료 후 `/forge-flow:verify`"
  - verified/tested → "`/forge-flow:complete`로 마무리하세요"
  - 기타 → 해당 phase에 맞는 스킬 호출 안내
- "추가 요구사항 반영" → 기존 design 파일이 존재하면 로드하여 수정. design 파일이 없으면 (clarify 도중 중단된 경우) 새로 작성. phase를 `"clarifying"`으로 되돌리고 1단계부터 진행 (기존 AC 유지 + 추가분 반영)
  - **주의**: phase가 `"implementing"` 이후였다면 경고 표시: "이미 구현된 코드가 있습니다. 추가 요구사항 반영 시 기존 구현과 충돌할 수 있습니다. `git diff`로 현재 변경 사항을 확인하세요."
  - **탐색 결과 재사용 (AC-14)**: verify FAIL 또는 추가 요구사항으로 clarify가 재실행되는 경우, 기존 탐색 결과가 design 문서의 영향 범위·기존 패턴에 이미 반영되어 있으면 **탐색 에이전트팀 재호출을 생략**. 요구사항 변경이 기존 영향 범위를 벗어날 때만 1단계 4-a 판단을 재적용하여 팀 재호출 여부 결정.
- "새 작업 시작" → 이전 상태 파일의 design 파일 + `.review.md` 삭제. 이전 상태 파일 삭제. 새 작업으로 진행

**B. 다른 세션의 미완료 작업 존재** (현재 세션 바인딩 없음)

```
question: "미완료 작업이 있습니다. 어떻게 진행할까요?"
header: "미완료 작업 발견"
options:
  - label: "{task_id_1} ({phase}) — 이어서 진행"
    description: "이 작업을 현재 세션에서 이어서 진행합니다"
  ...추가 작업이 있으면 각각 선택지로...
  - label: "새 작업 시작"
    description: "미완료 작업을 보관하고 새 작업을 시작합니다"
multiSelect: false
```

- 기존 작업 선택 → 해당 상태 파일의 `session_id`를 현재 세션으로 갱신하고 `stop_count`를 0으로 리셋 → A의 "이어서 진행"과 동일하게 처리
- "새 작업 시작" → 새 작업으로 진행 (미완료 작업은 그대로 유지)

> 미완료 작업이 5건 이상이면 최근 5건만 선택지로 표시하고, 나머지는 "그 외 {N}건은 `.forge-flow/state/`에서 확인하세요"로 안내합니다.

**C. orphan design 파일만 존재** (state 없는 design)

```
question: "상태 파일 없는 design 파일이 있습니다. 어떻게 진행할까요?"
header: "orphan design 발견"
options:
  - label: "{design_name} — 기반으로 추가/보완 작업"
    description: "이 design을 기반으로 새 워크플로를 시작합니다"
  ...추가 파일이 있으면 각각 선택지로...
  - label: "새 작업 시작 (orphan 삭제)"
    description: "orphan 파일을 삭제하고 새 작업을 시작합니다"
multiSelect: false
```

- 기존 design 선택 → 새 상태 파일 생성 (task_id = design 파일명, phase = "clarifying") → 기존 design 기반으로 1단계 진행 (추가/수정할 부분 질문)
- "새 작업 시작" → orphan design 파일 + `.review.md` 삭제 → 새 작업으로 진행

**D. 기존 작업 없음** → 새 작업 생성 (아래 실행 흐름으로 직행)

> 삭제된 design 파일은 git 히스토리에서 복원 가능합니다.

## 탐색 에이전트팀 (공통)

> **목적**: 정보 수집. clarify·plan의 무거운 codebase 탐색을 메인에서 격리하여 컨텍스트 오염(345K+ 토큰 케이스) 방지.
> **review 계열과 구분**: review-req/review-plan/verify/test의 교차검증 팀과 달리 "정보 수집"이 목적. 역할·프롬프트·기대 산출물이 다르다.
> **호출 조건**: **규모 무관 항상 호출**. 단순 프롬프트만으로 규모를 추정할 수 없으므로 (닭-달걀 문제), 메인이 직접 탐색하지 않고 항상 팀에 위임하여 요약만 받는다. 팀의 결과 요약으로 메인이 S/M/L을 판정한다.

### 팀 구조

| 역할 | 책임 |
|------|------|
| **Explorer** | 파일·심볼·경로 탐색. grep/Glob/Read로 관련 파일 식별 |
| **Analyzer** | 영향 범위 + 기존 패턴(네이밍·에러처리·테스트 패턴 등) 분석 |

### Spawn 절차

기존 review 팀과 동일한 체인:

1. **`TeamCreate`**: `{"team_name": "explore-{task_id}-{단계}", "description": "탐색 팀 - {단계}"}`
2. **`TaskCreate`** 2건 (Explorer 작업, Analyzer 작업)
3. **`Agent`**: `subagent_type: "Explore"`, `team_name`, `name: "explorer"`/`"analyzer"`, `model: "sonnet"`
4. 팀원 메시지 자동 수신 → 요약 종합 → `SendMessage(shutdown_request)` → `TeamDelete`

### 폴백 정책 (팀 호출 실패 시)

| # | 실패 모드 | 처리 |
|---|----------|------|
| (a) | TeamCreate 오류 | 전체 폴백 → 메인 직접 탐색 |
| (b) | 팀원 spawn 실패 | 전체 폴백 |
| (c) | 타임아웃 (기본 5분) — **양쪽 미응답** | 전체 폴백 |
| (c') | 타임아웃 — **한쪽이 유효 결과 반환** | 부분 실패 — (d) 규칙 적용 |
| (d) | 부분 실패 (Explorer 또는 Analyzer 한쪽만 실패) | 성공한 쪽 결과 유지, 실패한 쪽만 메인이 폴백 수행 |
| (e) | 빈 요약 리턴 | 전체 폴백 |

**폴백 공통 규칙**:
- (b) 폴백 경로: 메인이 직접 grep/Glob/Read로 탐색
- (c) 포맷 유지: 메인이 탐색 후 위 "단계별 요약 포맷"과 동일하게 정리하여 후속 단계 흐름 유지
- (d) 사용자 고지: `탐색 팀 실패로 메인 직접 탐색 진행 — 컨텍스트 사용량 증가 가능` 한 줄 표시 (**자동 진행, 확인 불필요**).
  - 이는 clarify/plan 내부의 **서브 레벨** 탐색 실패 처리이며, CLAUDE.md 워크플로 2.5단계(review-req → plan 경계)의 사용자 확인과 별개 레이어다.
- (e) state 갱신: 상태 파일 `agent_team.members[].status`에 `"fallback"` 리터럴 기록. phase 전이는 정상 진행.

### state 스키마 확장

기존 `agent_team` 필드(`enabled`, `members`)에 **값 확장만** 추가:
- `members[].status`에 `"fallback"` 리터럴 허용 (기존 `"active"/"completed"`에 병렬 추가)
- 재개 인프라 강화(제외 범위)에 해당하지 않음 — 새 필드 도입이 아닌 값 확장 수준.

## 실행 흐름

### 1단계: 프롬프트 분석 + 프로젝트 컨텍스트

1. 사용자 프롬프트에서 모호한 부분, 누락된 정보 식별
2. CLAUDE.md 읽기 — 프로젝트 설정, 빌드 명령, 기존 패턴 파악
3. **rework-log 참조**: `.forge-flow/rework-log.md` 존재 시 `[코드]`, `[요구사항]` 차원의 항목을 스캔하여 이번 작업과 관련된 과거 REWORK 패턴이 있는지 확인. 관련 패턴 발견 시 AC 또는 주의사항에 반영
4. **영향 범위 사전 조사 — 탐색 팀에 무조건 위임**

   규모 무관 **항상** `## 탐색 에이전트팀 (공통)` 섹션의 Spawn 절차대로 팀 호출. "clarify 1단계 (개략)" 출력 포맷으로 요약을 리턴 받아 이후 단계에서 재사용.
   - LSP 사용 가능 환경이면 Explorer가 LSP 기반 참조·타입 확인 수행, 없으면 grep + Glob 폴백
   - **메인은 직접 grep/Glob/Read 하지 않는다**. 사용자 프롬프트만으로는 규모를 판정할 수 없고, 메인이 탐색을 시작하면 컨텍스트가 오염된다 (345K+ 토큰 케이스의 원인).
   - 팀이 돌려준 요약(`[파일 목록]`, `[영향 범위]`, `[리스크/주의사항]`)을 근거로 4단계에서 S/M/L 예비 판정을 수행한다.

   > **1단계 탐색 결과 재사용**: 위에서 받은 요약은 **3단계 질문 생성**(모호 지점 식별)과 **4단계 예비 규모 판정**에 재사용한다. 중복 탐색을 피해 컨텍스트 절약.

### 1-B단계: 작업 위치 선택

프롬프트 분석 후, 이번 작업의 작업 위치를 선택합니다. **모든 clarify에서 항상 실행합니다.**

1. **git repo 감지**: `git rev-parse --show-toplevel`로 CWD 내 git repo root를 감지합니다.
   - 단일 repo: CWD 자체가 repo root
   - 중첩 repo (safenow/safenow-vasanta 구조): 하위 디렉토리에 별도 `.git/`이 있으면 repo 경로를 사용자에게 확인
2. **기존 워크트리 확인**: `git -C {repo} worktree list`로 현재 워크트리 목록을 확인합니다.

**AskUserQuestion 호출**:

기존 워크트리가 **있을 때** (3개 선택지):
```
question: "이번 작업의 작업 위치를 선택하세요."
header: "작업 위치"
options:
  - label: "현재 디렉토리 (Recommended)"
    description: "기존 코드 디렉토리에서 작업합니다"
  - label: "새 워크트리 생성"
    description: "별도 워크트리를 생성하여 격리된 환경에서 작업합니다"
  - label: "기존 워크트리 선택"
    description: "이미 생성된 워크트리에서 작업합니다"
multiSelect: false
```

기존 워크트리가 **없을 때** (2개 선택지):
```
question: "이번 작업의 작업 위치를 선택하세요."
header: "작업 위치"
options:
  - label: "현재 디렉토리 (Recommended)"
    description: "기존 코드 디렉토리에서 작업합니다"
  - label: "새 워크트리 생성"
    description: "별도 워크트리를 생성하여 격리된 환경에서 작업합니다"
multiSelect: false
```

#### "현재 디렉토리" 선택 시

state 파일에 `work_dir`을 기록하지 않습니다 (기존 동작 유지). 2단계로 진행.

#### "새 워크트리 생성" 선택 시

1. **base branch 입력**: `AskUserQuestion`으로 base branch를 질문합니다.
   ```
   question: "워크트리의 base branch를 입력하세요. (예: develop, main)"
   header: "Base branch"
   options:
     - label: "main"
       description: "main 브랜치 기반"
     - label: "develop"
       description: "develop 브랜치 기반"
   multiSelect: false
   ```
   > 사용자가 Other로 직접 입력 가능.

2. **워크트리 이름**: task_id를 기본값으로 사용 (사용자가 변경 가능).

3. **워크트리 생성 실행**:
   ```bash
   git -C {repo} worktree add ../worktrees/{name} {base_branch}
   ```
   - 성공 시 → `worktrees/` 디렉토리를 프로젝트 루트의 `.gitignore`에 추가 (이미 있으면 스킵)
   - 실패 시 (브랜치 미존재, 경로 중복 등) → 오류 메시지 표시 후 1-B단계 처음으로 돌아가 선택지 재표시

4. **의존성 설치**: CLAUDE.md `## 빌드 명령`이 설정되어 있으면 워크트리 내에서 의존성 설치 실행 안내:
   ```
   워크트리가 생성되었습니다: worktrees/{name}/
   의존성 설치가 필요할 수 있습니다: cd worktrees/{name} && {빌드 명령}
   ```
   > 빌드 명령이 미설정이면 이 안내를 생략합니다.

5. **state 기록**: 생성 성공 확인 후에만 state 파일에 기록합니다:
   ```json
   { "work_dir": "worktrees/{name}", "base_branch": "{base_branch}" }
   ```

#### "기존 워크트리 선택" 선택 시

1. `git -C {repo} worktree list`의 결과를 파싱하여 AskUserQuestion 선택지로 표시합니다.
   - 메인 워크트리(bare)는 제외
   - 각 항목: `{경로} ({branch})`
2. 선택한 워크트리의 상대 경로를 state `work_dir`에 기록합니다.
3. 해당 워크트리의 branch에서 base branch를 추론합니다 (추론 불가 시 사용자에게 질문).

### 2단계: 요구사항 재진술

받은 요구사항을 내가 이해한 대로 다시 서술한 뒤, `AskUserQuestion`으로 확인합니다.

```
[내가 이해한 요구사항]
<원본 요구사항을 자신의 언어로 1~3문장으로 재서술>
```

**AskUserQuestion 호출**:
```
question: "위 이해가 맞나요?"
header: "요구사항"
options:
  - label: "맞습니다"
    description: "이 이해로 진행합니다"
  - label: "부분 수정"
    description: "대체로 맞지만 일부 수정이 필요합니다"
  - label: "다시 설명"
    description: "이해가 많이 다릅니다. 다시 설명하겠습니다"
multiSelect: false
```

- "맞습니다" → 3단계 진행
- "부분 수정" / "다시 설명" / Other → 수정된 내용으로 다시 재진술 → 재확인

> **추가 codebase 조사가 필요한 경우 (AC-11)**: "부분 수정"·"다시 설명" 응답에 따라 추가 조사가 필요하면 **규모 무관 `## 탐색 에이전트팀 (공통)`의 "clarify 1단계 (개략)" 포맷으로 팀 재호출**. 메인이 직접 grep/Read 하지 않는다.

### 3단계: 핵심 질문 (최대 5개)

요구사항 이해 후, 불명확한 부분만 집중 질문합니다. **5개를 넘기지 않습니다.**

| 항목 | 질문 예시 |
|------|---------|
| 완료 조건 | "어떤 상태가 되면 이 작업이 '완료'인가요?" |
| 변경 범위 | "어느 서비스/화면까지 변경이 필요한가요?" |
| 해석 분기 | "A 방식과 B 방식 중 어느 쪽인가요?" |
| 예외/엣지케이스 | "기존 데이터는 어떻게 처리하나요?" |
| 우선순위 | "먼저 해야 할 부분이 있나요?" |

**AskUserQuestion 활용**: 선택지가 명확한 질문은 `AskUserQuestion`으로 구조화합니다.
- 1회 호출에 최대 4개 질문까지 묶을 수 있음
- 선택지가 있는 질문 → `AskUserQuestion` (예: 해석 분기, 우선순위)
- 자유 서술이 필요한 질문 → 텍스트로 질문

```
# 예시: 해석 분기 + 우선순위를 한 번에 질문
questions:
  - question: "인증 방식을 어떻게 구현할까요?"
    header: "인증 방식"
    options:
      - label: "JWT 토큰"
        description: "Stateless, 확장성 우수"
      - label: "세션 기반"
        description: "서버 측 상태 관리, 기존 패턴"
    multiSelect: false
  - question: "먼저 구현할 부분이 있나요?"
    header: "우선순위"
    options:
      - label: "백엔드 먼저"
        description: "API 완성 후 프론트엔드"
      - label: "프론트엔드 먼저"
        description: "UI 완성 후 API 연동"
      - label: "동시 진행"
        description: "BE/FE 병렬 작업"
    multiSelect: false
```

> **추가 codebase 조사가 필요한 경우 (AC-2)**: 3단계 질문·답변 과정에서 "현재 ~가 어떻게 구현되어 있는지" 같은 추가 확인이 필요하면, **규모 무관** 메인이 직접 grep/Read 하지 말고 `## 탐색 에이전트팀 (공통)`의 "clarify 3단계 (질문 중 추가 조사)" 포맷으로 팀을 재호출한다.
>
> **사용자 범위 확장 시 (AC-15)**: 3단계 중 사용자가 "그리고 ~도 추가" 같은 범위 확장 입력을 넣으면 **규모 무관** "clarify 1단계 (개략)" 포맷으로 팀을 재호출한다.

### 4단계: 예비 규모 판정

요구사항과 영향 범위를 기반으로 예비 규모를 판정합니다.

| 판정 | 근거 | 조치 |
|------|------|------|
| **명백한 L** | 사용자가 "대규모 리팩토링", "아키텍처 변경" 명시 / 영향 모듈 3개 이상 | 사용자에게 에이전트팀 즉시 제안 가능 |
| **S/M/불확실** | 위 경우가 아닌 모든 것 | M으로 간주, plan 이후 확정 판정 |

> 이 시점의 판정은 예비입니다. 확정 판정은 plan에서 변경 파일 목록 기반으로 이루어집니다.
>
> **사용자 범위 확장 시 (AC-15)**: 4단계 도중 사용자가 범위를 확장하면 **규모 무관** "clarify 1단계 (개략)" 포맷으로 팀을 재호출한다.
>
> **S 규모 확정 후에도 탐색은 팀 경유**: S 예비 판정 시에도 이후 단계(plan 1단계 포함)의 탐색은 계속 팀에 위임한다. 메인이 직접 grep/Read 하지 않는 원칙은 규모와 무관하다.

### 4-B단계: 검증 관점 결정

AC 확정 후, 이번 작업에 적용할 검증 관점을 결정합니다.

1. **CLAUDE.md `### 프로젝트 검증 관점` 섹션 확인**: 있으면 각 관점의 `적용 단계`와 `설명`을 읽음. 없으면 기본 관점만 사용
2. **AC와 프로젝트 관점 매칭**: 각 프로젝트 관점의 설명과 AC 항목을 대조하여 관련 있는 관점 활성화
3. **작업 특화 관점 도출**: AC에 프로젝트 관점으로 커버되지 않는 특수 요구(데이터 마이그레이션, 호환성 등)가 있으면 작업 특화 관점 추가
4. **5단계 스펙에 포함**: 활성화된 검증 관점을 스펙에 표시하여 사용자가 확인·조정 가능

> 검증 관점 결정은 자동 도출 후 5단계 스펙 확정 시 사용자가 확인합니다 (AskUserQuestion 추가 없음). 사용자가 "수정 필요"를 선택하면 검증 관점도 조정 가능.

### 5단계: 스펙 확정 + design 파일 생성

답변을 바탕으로 확정된 스펙을 출력하고 동의를 받습니다.

```
[확정된 요구사항 스펙]

목적: <한 문장>
완료 조건: <검증 가능한 조건>
변경 범위:
  - <서비스/파일/컴포넌트>
제외 범위: <이번 작업에서 하지 않는 것>
주의사항: <알려진 제약이나 위험>
예비 규모: M (또는 L — 근거 명시)
검증 관점: 기본(완전성, 실현성) + <AC 분석 기반 활성화된 프로젝트 관점>
```

**AskUserQuestion 호출**:
```
question: "이 스펙으로 진행할까요?"
header: "스펙 확정"
options:
  - label: "승인 (Recommended)"
    description: "이 스펙으로 design 문서를 생성하고 진행합니다"
  - label: "수정 필요"
    description: "일부 항목을 수정한 뒤 다시 확인합니다"
  - label: "처음부터 다시"
    description: "요구사항을 다시 정리합니다"
multiSelect: false
```

- "승인" → `.forge-flow/design/{작업명}.md` 파일 생성
- "수정 필요" / Other → 수정 후 재확인
- "처음부터 다시" → 1단계부터 재실행

> **동일 파일명 충돌 방지**: `.forge-flow/design/{작업명}.md`가 이미 존재하고 다른 상태 파일이 참조 중이면 `{작업명}-2.md` 등으로 자동 넘버링.

### .forge-flow/design/ 파일 포맷

```markdown
# {작업명}

## 요구사항
- 목적: <한 문장>
- 완료 조건: <검증 가능한 조건>

## 인수 조건 (AC)
- [ ] AC-1: <구체적이고 검증 가능한 조건>
- [ ] AC-2: <정상 시나리오: 입력 → 기대 출력>
- [ ] AC-3: <경계값/예외: 조건 → 기대 동작>

## 영향 범위
- 변경 대상: <파일/모듈 목록 (개략)>
- 영향 받는 기존 기능: <기존 로직 중 영향 받는 것>

## 제외 범위
- <이번 작업에서 하지 않는 것>

## 주의사항
- <알려진 제약이나 위험>

## 검증 관점
- 기본: 완전성, 실현성
- 프로젝트: (CLAUDE.md `### 프로젝트 검증 관점`에서 AC와 관련된 관점 활성화, 없으면 생략)
- 작업 특화: (AC 분석에서 도출된 추가 관점, 없으면 생략)

## 규모 판정
- 예비 (clarify): M — <근거>
- 확정 (plan): (plan에서 기입)

## 검증 설정
(plan 4단계에서 기입: 검증 강도, 수렴 라운드, 테스트 방식)

## 구현 계획
(plan에서 작성)

## 검수 결과
(review-req, review-plan, verify, test에서 기입)
```

> 검수 상세 이력은 `.forge-flow/design/{task_id}.review.md`에 별도 기록됩니다 (design 파일 비대화 방지).

### AC 작성 가이드

AC는 verify의 핵심 판단 기준입니다. **검증 가능한** 형태로 작성합니다.

| 좋은 AC | 나쁜 AC |
|---------|---------|
| "POST /auth/google → 200 + JWT 토큰 반환" | "구글 로그인이 잘 되어야 한다" |
| "비밀번호 8자 미만 입력 시 400 에러 + 메시지" | "비밀번호 검증 필요" |
| "`./gradlew test` 전체 통과" | "테스트가 통과해야 한다" |

## 상태 파일 생성

design 파일 생성 완료 시 `.forge-flow/state/{작업명}.json` 상태 파일을 생성합니다:

```json
{
  "task_id": "{작업명}",
  "session_id": "${CLAUDE_SESSION_ID}",
  "phase": "clarifying",
  "scale": null,
  "design_file": ".forge-flow/design/{작업명}.md",
  "stop_count": 0,
  "rework_counts": {},
  "started_at": "<현재 시각 ISO 8601>",
  "work_dir": null,
  "base_branch": null
}
```

> `task_id`는 design 파일명(확장자 제외)과 동일합니다. 상태 파일명도 `{task_id}.json`으로 1:1 매칭됩니다.
> `session_id`는 현재 세션을 바인딩합니다. 다른 세션에서 이어서 진행 시 이 값이 갱신됩니다.
> `stop_count`는 phase 전진 전이 시 0으로 리셋합니다. `rework_counts`는 각 스킬별 독립 카운터(`rework_counts.verify`, `rework_counts.test` 등)로 PASS 시 해당 스킬만 리셋됩니다.
> `work_dir`은 1-B단계에서 워크트리를 선택한 경우에만 기록됩니다. `null`이면 현재 디렉토리에서 작업 (기존 동작 유지).
> `base_branch`는 워크트리 생성 시 base branch를 기록합니다. complete에서 merge 대상으로 참조됩니다.

## 완료 후 다음 단계

clarify의 모든 작업이 완료되면(사용자 스펙 확정 동의 + design 파일 생성), 사용자의 추가 입력 없이 **즉시 `/forge-flow:review-req`를 호출**하여 요구사항 교차검증을 진행합니다.
