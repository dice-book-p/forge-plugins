verify(코드 검증) 통과 후, 실제 런타임에서 동작을 검증합니다.
브라우저 자동화로 UI/UX를 테스트하고, API 엔드포인트를 호출하여 결과를 확인합니다.

## 선행 조건 검사

실행 전 반드시 확인:
1. 현재 세션에 바인딩된 상태 파일 탐색 → `.forge-flow/state/`에서 `session_id`가 현재 세션(`${CLAUDE_SESSION_ID}`)과 일치하는 `{task_id}.json` 파일 탐색 → 없으면: "워크플로가 시작되지 않았습니다. `/forge-flow:clarify`로 시작하세요."
2. phase가 `"verified"` 또는 `"testing"`인지 확인 → 아니면: "현재 `{phase}` 단계입니다. verify를 먼저 통과하세요."
3. `design_file`이 존재하는 파일 경로인지 확인 → 없으면: "설계 문서를 찾을 수 없습니다."

## 상태 파일 갱신

실행 시작 시:
```json
{ "phase": "testing" }
```

> `rework_count`는 **최초 test 진입 시에만** 0으로 초기화합니다. REWORK 후 재진입 시에는 기존 `rework_count`를 유지합니다 (3회 에스컬레이션 추적을 위해).
> 최초 진입 판단: 이전 phase가 `"verified"`이면 최초 진입 → `rework_count: 0`. 이전 phase가 `"testing"`이면 재진입 → 유지.

## 검증 강도 확인

design 문서의 `## 검증 강도` 섹션을 확인합니다. 검증 강도에 따라 테스트 수준이 결정됩니다:

- **경량 (0)**: 스킵 조건을 완화하여 적용. 스킵 조건에 해당하면 즉시 스킵.
- **표준 (1)**: 단일 테스터 (기본).
- **강화 (2+)**: 테스트 팀 구성 (테스트 도메인 분리).

> 검증 강도가 미설정이면 규모 기반 기본값을 적용합니다 (S=0, M=1, L=2).

## 스킵 조건

아래 **모두** 해당하면 test를 스킵하고 `/forge-flow:complete`를 호출합니다:
- 규모가 S
- design 문서에 UI/UX 관련 AC가 없음
- CLAUDE.md에 `## 테스트 환경` 섹션이 없음

스킵 시:
```
[test 스킵] S 규모 + UI/UX AC 없음 → 실행 테스트 불필요
```

## 테스트 범위 분석

design 문서의 AC 항목을 분석하여 테스트 유형을 결정합니다:

| AC 유형 | 테스트 방식 | 예시 |
|---------|-----------|------|
| UI 동작 | 브라우저 자동화 | "버튼 클릭 시 모달이 열린다" |
| 페이지 표시 | 브라우저 자동화 | "목록에 10개 항목이 표시된다" |
| API 응답 | API 직접 호출 | "POST /api/users → 201 반환" |
| 데이터 처리 | API + DB 확인 | "주문 생성 시 재고가 감소한다" |
| 폼 유효성 | 브라우저 자동화 | "이메일 형식 오류 시 에러 표시" |

## 테스트 환경 준비

테스터 프롬프트의 `## 테스트 환경` 섹션을 채울 때:
1. 글로벌/프로젝트 CLAUDE.md의 브라우저 테스트 관련 지침을 확인
2. 테스트 도구 선택 규칙이 있으면 해당 내용을 Read하여 프롬프트에 포함
3. 프로젝트 CLAUDE.md의 `## 테스트 환경` 섹션이 있으면 함께 포함

## 테스트 실행 방식

### 에이전트팀 vs 서브에이전트 결정 (필수 확인)

아래 의사결정 트리를 **순서대로** 확인하여 방식을 결정합니다:

```
1. 스킵 조건 해당 → test 스킵 (/forge-flow:complete 호출)
2. CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 설정됨?
   ├─ YES → 에이전트팀 사용 (기본)
   │   └─ 도메인 2개 이상 → 팀원 복수 구성
   │   └─ 도메인 1개 → 팀원 1명 구성
   └─ NO  → 서브에이전트 폴백
3. 검증 강도 2+ → 도메인 무관 복수 구성 강제
```

> **에이전트팀이 기본입니다.** `setup-workflow`에서 기본 활성화되므로, 별도 비활성화하지 않는 한 항상 에이전트팀을 사용합니다.

### 테스트 팀 (에이전트팀)

#### 테스트 팀 구성 분석

design 문서의 AC를 분석하여 독립 테스트 도메인을 식별합니다:

| 분리 기준 | 예시 |
|----------|------|
| **도메인 분리** | 사용자단 UI 테스트 / 관리자단 UI 테스트 |
| **유형 분리** | 브라우저 UI 테스트 / API 통합 테스트 |
| **시나리오 분리** | 정상 플로우 / 에러·엣지케이스 플로우 |

#### 테스트 팀 구성 보고


#### 에이전트팀 spawn 절차

1. **`TeamCreate` 호출**: `{ "team_name": "test-{task_id}", "description": "테스트 팀" }`

4. 팀원 메시지 자동 수신 → 숙의 → `SendMessage`로 종료 → `TeamDelete`

테스트 도메인별로 팀원을 동적으로 구성합니다 (`model: "sonnet"`):

## 결과 종합

**단일 테스터**:
- 통과 → 작업 완료
- 실패 → 실패 항목 정리 → 수정 후 재테스트

**테스트 팀 — 숙의(Deliberation)** (리더가 종합):
1. 모든 테스터의 결과 수집
2. AC × 테스터 매트릭스 작성
3. **불일치 식별**: 동일 AC를 복수 테스터가 검증한 경우, 판정이 다른 항목 목록화
4. **갈등 분석** (불일치 항목이 있을 때):
   - 테스트 환경 차이인지, 실제 결함인지 판별
   - 근거: 스크린샷/로그 증거 비교
5. **판정 근거 작성**: 최종 판정에 대해 "왜 이 결정을 내렸는지" 기록
6. 실패한 테스터만 재실행 가능 (전체 재실행 불필요)
7. 최종 판정 도출

**숙의 결과 보고 형식**:
```
[숙의 결과]
  일치 항목: AC-1(PASS), AC-5(PASS), ...
  불일치 항목:
    - AC-3: 테스터-UI(PASS) vs 테스터-에러(FAIL)
      → 채택: FAIL — 에러 경로에서 모달 미표시 (스크린샷 참조)
  판정 근거: AC-3 에러 처리 미구현으로 REWORK 판정
```

## 품질 게이트

| 등급 | 기준 | 조치 |
|------|------|------|
| **통과**(PASS) | 전 테스트 항목 통과 | 작업 완료 |
| **주의**(CONCERNS) | 경미한 UI 이슈 (스타일 미세 차이 등) | `AskUserQuestion`으로 판단 위임 |
| **재작업**(REWORK) | AC 미충족 (기능 미동작, API 에러 등) | 해당 부분 수정 → 재테스트 |
| **실패**(FAIL) | 근본적 문제 (페이지 미로딩, 서버 에러 등) | 이전 단계부터 재검토 |

> **재작업 연속 3회 시 실패로 에스컬레이션.**

**주의 판정 시 AskUserQuestion 호출**:
```
question: "경미한 UI 이슈가 발견되었습니다. 어떻게 진행할까요?"
header: "주의"
options:
  - label: "수용 — 진행 (Recommended)"
    description: "경미한 이슈를 인지하고 다음 단계로 진행합니다"
  - label: "수정 후 재테스트"
    description: "이슈를 수정한 뒤 test를 다시 실행합니다"
multiSelect: false
```

**재작업 처리 흐름**:
1. 실패한 테스트 항목과 근거(스크린샷/로그)를 사용자에게 보고
2. **rework-log 기록** (verify SKILL.md의 "Rework Log 관리" 규칙에 따름)
3. 상태 파일의 `rework_count`를 +1 (`rework_count` ≥ 3이면 실패로 에스컬레이션)
4. phase를 `"implementing"`으로 되돌림
5. 수정 완료 후 `/forge-flow:verify` → `/forge-flow:test` 재실행

## 테스트 결과 기록

design 문서의 `## 검수 결과`, `## 검수 이력` 섹션에 기록:

```markdown
## 검수 결과
- verify: PASS (2026-03-17)
- test: PASS (2026-03-17)

## 검수 이력
### test
- #1 REWORK: 사용자 목록 페이지에서 페이지네이션 미동작
  → 수용: PageableResolver 파라미터 바인딩 수정
- #2 PASS
```

## 상태 파일 갱신 (완료 시)

PASS 시:
```json
{ "phase": "tested", "stop_count": 0, "rework_count": 0 }
```

REWORK 시:
```json
{ "phase": "implementing", "rework_count": +1 }
```

FAIL 시:
```json
{ "phase": "clarifying" }
```

## 완료 후 다음 단계

test PASS 후 → 사용자의 추가 입력 없이 **즉시 `/forge-flow:complete`를 호출**합니다.

> `/forge-flow:complete`에서 커밋 여부 확인, design 문서 처리, 워크플로 파일 정리를 수행합니다.
