plan에서 작성된 구현 계획이 요구사항과 일치하는지 에이전트팀 외부 검수로 검증합니다.

## 실행 조건

- **L 규모**: 필수 실행
- **M 규모**: 아래 조건 중 하나라도 해당 시 실행
  - 새 외부 의존성 도입
  - API 계약 변경 (요청/응답 스키마)
  - DB 스키마 변경
  - 3개 이상 모듈에 영향
  - 기존 프로젝트에 없던 패턴 도입
- **S 규모**: 스킵

## 선행 조건 검사

실행 전 반드시 확인:
1. 현재 세션에 바인딩된 상태 파일 탐색 → `.forge-flow/state/`에서 `session_id`가 현재 세션(`${CLAUDE_SESSION_ID}`)과 일치하는 `{task_id}.json` 파일 탐색 → 없으면: "워크플로가 시작되지 않았습니다. `/forge-flow:clarify`로 시작하세요."
2. design 문서의 `## 구현 계획` 섹션이 작성되어 있는지 확인 → 없으면: "구현 계획이 없습니다. `/forge-flow:plan`을 먼저 실행하세요."

## 상태 파일 갱신

실행 시작 시:
```json
{ "phase": "reviewing-plan", "stop_count": 0, "rework_count": 0 }
```

## 실행 흐름

### 1단계: 에이전트팀 구성

design 문서에서 요구사항(AC)과 구현 계획을 로드한 뒤, 에이전트팀 외부 검수를 수행합니다.

**규모별 팀 크기**:

| 규모 | 팀원 수 | 구성 |
|------|---------|------|
| **M** | 1명 | 구현 가능성 + AC 커버리지 통합 검증 |
| **L** | 2명 | 구현 가능성 검증자 / AC 커버리지 검증자 |

**에이전트팀 spawn 절차** (review-req/verify 패턴 따름):

1. **`TeamCreate` 호출**: `{ "team_name": "review-plan-{task_id}", "description": "설계 검증 팀" }`
2. **`TaskCreate`로 검증 작업 생성**
3. **`Agent` 도구 + `team_name` + `name` 파라미터로 팀원 spawn** (model 생략 → 세션 모델 상속)
4. 팀원 메시지 자동 수신 → 숙의 → `SendMessage`로 종료 → `TeamDelete`

**검증 팀 구성 보고**:
```
[검증 팀 구성 — 에이전트팀]
  team: review-plan-{task_id}
  검증자 A: 구현 가능성 — 실제 코드 탐색 기반 실현성 검증
  검증자 B: AC 커버리지 — AC↔구현파일 1:1 매핑 검증 (L규모만)

검증 팀을 spawn합니다.
```

**검증자 프롬프트 템플릿** (model 생략 → 세션 모델 상속):
```
당신은 {프로젝트명}의 독립 설계 검증자입니다.
검증 관점: {담당 관점}

## design 문서
{design 문서 전문}

## 검증 항목
### 구현 가능성 검증 (해당 시)
- 구현 계획의 변경 대상 파일이 실제로 존재하는가? (Glob/Grep으로 탐색)
- '따를 기존 패턴'의 근거 파일이 존재하고 해당 패턴이 맞는가?
- 변경 순서가 의존성 기반으로 타당한가?
- 리스크 항목에 대응 방안이 있는가?

### AC 커버리지 검증 (해당 시)
- 각 AC가 구현 계획의 어떤 파일/변경에서 충족되는가? 1:1 매핑
- 변경 전파 체인에 누락된 파일이 없는가?

## 응답 형식
각 항목에 대해:
- PASS: <근거 한 줄>
- FAIL: <구체적 문제점 + 수정 제안>

## 종합 판정
PASS / CONCERNS / REWORK / FAIL + 근거
```

> **핵심 원칙**: 메인 세션(plan 작성자)의 판단 결과를 검증 팀원에게 전달하지 않음 (독립 판단 보장).

### 2단계: 결과 종합 — 숙의(Deliberation) (리더)

verify 패턴과 동일:
1. 모든 검증자의 메시지 자동 수신 대기
2. **불일치 식별**: 검증자 간 판정이 다른 항목 목록화
3. **갈등 분석** (불일치 항목이 있을 때):
   - 각 불일치에 대해 "어느 관점이 더 타당한가?" 추론
   - 근거: 코드 증거(파일:라인), AC 원문, 패턴 기준 중 어느 것에 기반하는지 명시
4. **판정 근거 작성**: 최종 판정에 대해 "왜 이 결정을 내렸는지" 기록
5. 최종 판정 도출

**숙의 결과 보고 형식**:
```
[숙의 결과]
  일치 항목: 구현 가능성(PASS), AC 커버리지(PASS), ...
  불일치 항목:
    - 변경 순서: 검증자-가능성(FAIL) — 의존성 역전 감지 (fileA → fileB)
      → 채택: FAIL
  판정 근거: 변경 순서 역전으로 REWORK 판정
```

### 3단계: 품질 게이트

4단계 품질 게이트:

| 등급 | 기준 | 조치 |
|------|------|------|
| **통과**(PASS) | 전 항목 문제 없음 | 구현으로 진행 |
| **주의**(CONCERNS) | 경미한 이슈 (순서 조정 등) | `AskUserQuestion`으로 판단 위임 |
| **재작업**(REWORK) | AC 누락, 전파 체인 미반영 등 | plan 수정 → 재검수 |
| **실패**(FAIL) | 근본적 설계 문제 | 사용자에게 재검토 요청 → clarify부터 재진행 |

> **재작업 연속 3회 시 실패로 에스컬레이션.**

**주의 판정 시 AskUserQuestion 호출**:
```
question: "경미한 이슈가 발견되었습니다. 어떻게 진행할까요?"
header: "주의"
options:
  - label: "수용 — 진행 (Recommended)"
    description: "경미한 이슈를 인지하고 다음 단계로 진행합니다"
  - label: "수정 후 재검수"
    description: "이슈를 수정한 뒤 review-plan을 다시 실행합니다"
multiSelect: false
```

**재작업 처리 흐름**:
1. 문제점을 사용자에게 보고
2. 상태 파일의 `rework_count`를 +1 (`rework_count` ≥ 3이면 실패로 에스컬레이션)
3. phase를 `"planning"`으로 되돌림
4. 사용자가 구현 계획 수정
5. 수정 완료 후 다시 `/forge-flow:review-plan` 실행

### 4단계: 결과 기록

design 문서의 `## 검수 결과`, `## 검수 이력` 섹션에 기록:

```markdown
## 검수 결과
- review-plan: PASS (2026-03-13)

## 검수 이력
### review-plan
- #1 REWORK: AC-2에 대응하는 구현 파일 누락
  → 수용: UserService.java 추가
- #2 PASS
```

## 상태 파일 갱신 (완료 시)

PASS 시:
```json
{ "phase": "implementing", "stop_count": 0, "rework_count": 0 }
```

REWORK 시:
```json
{ "phase": "planning", "rework_count": +1 }
```
> `rework_count`는 REWORK 판정마다 +1. 3 이상이면 FAIL로 에스컬레이션.
> `stop_count`는 REWORK/FAIL 후퇴 시 리셋하지 않습니다.

## 완료 후 다음 단계

review-plan PASS 후, 사용자의 추가 입력 없이 **즉시 설계대로 구현을 시작**합니다.

에이전트팀이 활성화되어 있고 plan에서 팀 구성이 승인된 경우 → 팀원 spawn + 병렬 구현 시작 (plan의 "에이전트팀 동적 구성" 절차에 따름).
