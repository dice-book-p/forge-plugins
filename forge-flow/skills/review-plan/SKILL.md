plan에서 작성된 구현 계획이 요구사항과 일치하는지 검수합니다.

## 실행 조건

- **L 규모**: 필수 실행
- **M 규모**: 아래 조건 중 하나라도 해당 시 실행
  - 새 외부 의존성 도입
  - API 계약 변경 (요청/응답 스키마)
  - DB 스키마 변경
  - 3개 이상 모듈에 영향
  - 기존 프로젝트에 없던 패턴 도입
- **S 규모**: 스킵

## 선행 조건 검사

실행 전 반드시 확인:
1. 상태 파일 존재 → 없으면: "워크플로가 시작되지 않았습니다. `/forge-flow:clarify`로 시작하세요."
2. design 문서의 `## 구현 계획` 섹션이 작성되어 있는지 확인 → 없으면: "구현 계획이 없습니다. `/forge-flow:plan`을 먼저 실행하세요."

## 상태 파일 갱신

실행 시작 시:
```json
{ "phase": "reviewing-plan", "stop_count": 0, "rework_count": 0 }
```

## 실행 흐름

### 1단계: 검수 항목 확인

design 문서에서 요구사항(AC)과 구현 계획을 로드하여 대조합니다.

| 검수 항목 | 확인 내용 |
|----------|----------|
| AC ↔ 설계 1:1 대응 | 각 AC가 구현 계획의 어떤 파일/변경에서 충족되는가? |
| 변경 전파 체인 누락 | 이 변경이 영향을 미치는 파일이 계획에 모두 포함되었는가? |
| 기존 패턴 일관성 | 구현 계획의 `따를 기존 패턴` 테이블이 작성되었는가? 각 패턴의 근거 파일이 실제로 존재하며 해당 패턴이 맞는가? 구현 계획이 이 패턴을 따르도록 설계되었는가? |
| 변경 순서 타당성 | 의존성 기반 변경 순서가 맞는가? |
| 리스크 대응 | 식별된 리스크에 대응 방안이 있는가? |

### 2단계: 판정

4단계 품질 게이트:

| 등급 | 기준 | 조치 |
|------|------|------|
| **통과**(PASS) | 전 항목 문제 없음 | 구현으로 진행 |
| **주의**(CONCERNS) | 경미한 이슈 (순서 조정 등) | `AskUserQuestion`으로 판단 위임 |
| **재작업**(REWORK) | AC 누락, 전파 체인 미반영 등 | plan 수정 → 재검수 |
| **실패**(FAIL) | 근본적 설계 문제 | 사용자에게 재검토 요청 → clarify부터 재진행 |

> **재작업 연속 3회 시 실패로 에스컬레이션.**

**주의 판정 시 AskUserQuestion 호출**:
```
question: "경미한 이슈가 발견되었습니다. 어떻게 진행할까요?"
header: "주의"
options:
  - label: "수용 — 진행 (Recommended)"
    description: "경미한 이슈를 인지하고 다음 단계로 진행합니다"
  - label: "수정 후 재검수"
    description: "이슈를 수정한 뒤 review-plan을 다시 실행합니다"
multiSelect: false
```

**재작업 처리 흐름**:
1. 문제점을 사용자에게 보고
2. 상태 파일의 `rework_count`를 +1 (`rework_count` ≥ 3이면 실패로 에스컬레이션)
3. phase를 `"planning"`으로 되돌림
4. 사용자가 구현 계획 수정
5. 수정 완료 후 다시 `/forge-flow:review-plan` 실행

### 3단계: 결과 기록

design 문서의 `## 검수 결과`, `## 검수 이력` 섹션에 기록:

```markdown
## 검수 결과
- review-plan: PASS (2026-03-13)

## 검수 이력
### review-plan
- #1 REWORK: AC-2에 대응하는 구현 파일 누락
  → 수용: UserService.java 추가
- #2 PASS
```

## 상태 파일 갱신 (완료 시)

PASS 시:
```json
{ "phase": "implementing", "stop_count": 0, "rework_count": 0 }
```

REWORK 시:
```json
{ "phase": "planning", "rework_count": +1 }
```
> `rework_count`는 REWORK 판정마다 +1. 3 이상이면 FAIL로 에스컬레이션.
> `stop_count`는 REWORK/FAIL 후퇴 시 리셋하지 않습니다.

## 완료 후 다음 단계

review-plan PASS 후, 사용자의 추가 입력 없이 **즉시 설계대로 구현을 시작**합니다.

에이전트팀이 활성화되어 있고 plan에서 팀 구성이 승인된 경우 → 팀원 spawn + 병렬 구현 시작 (plan의 "에이전트팀 동적 구성" 절차에 따름).
