# forge-flow 사용자 가이드

## 개요

forge-flow는 Claude Code 워크플로 자동화 플러그인입니다.
요구사항 명확화부터 검수까지 품질 게이트를 자동으로 적용하여, 완성도 높은 개발 경험을 제공합니다.

```
clarify → review-req → plan → review-plan(조건부) → 구현 → verify → 커밋
```

---

## 1. 설치

### 1.1 플러그인 등록 (최초 1회)

마켓플레이스가 아직 등록되지 않았다면:

```bash
claude plugin marketplace add /path/to/claude-plugins
claude plugin install forge-flow
```

이후 모든 프로젝트에서 `/forge-flow:*` 스킬을 사용할 수 있습니다.

### 1.2 프로젝트 초기화 (프로젝트별 1회)

적용할 프로젝트 디렉토리에서 Claude Code를 열고:

```
/forge-flow:install
```

설치되는 항목:
- **CLAUDE.md** — 워크플로 규칙, 빌드 명령, 브랜치 전략 섹션 삽입
- **`.claude/hooks/`** — 3개 훅 스크립트 (종료 차단, 상태 알림, 위험 명령 차단)
- **`design/`** — 작업별 설계 문서 디렉토리
- **`.forge-flow/`** — 세션별 상태 파일 디렉토리 (`.gitignore`에 자동 추가)

기존 프로젝트는 빌드/테스트 명령, 브랜치 전략 등을 자동 감지합니다.
신규 프로젝트는 Q&A로 수집합니다.

### 1.3 설치 옵션

```
/forge-flow:install          # 신규 설치 또는 업그레이드 자동 감지
/forge-flow:install --update # 변경된 항목만 감지 → 선택 적용
/forge-flow:install --reset  # 전체 제거 → Q&A → 재설치
/forge-flow:install --purge  # 전체 제거만 (확인 후 실행)
```

## 2. 워크플로 사용법

### 2.1 작업 시작

새 작업은 `/forge-flow:clarify`로 시작합니다.

```
/forge-flow:clarify
```

또는 자연어로 작업을 요청하면 자동 트리거됩니다:
- "로그인 기능 추가해줘"
- "결제 API 버그 수정"
- "사용자 목록 페이지 리팩토링"

clarify가 수행하는 일:
1. 요구사항을 재진술하여 이해 확인
2. 최대 5개 핵심 질문으로 모호함 해소
3. 예비 규모 판정 (S/M/L)
4. `design/{작업명}.md` 설계 문서 생성
5. 수용 기준(AC) 작성 — 검증 가능한 형태로

### 2.2 요구사항 검수 (자동)

clarify 완료 후 자동으로 실행됩니다.

```
/forge-flow:review-req
```

서브에이전트가 독립적으로 설계 문서를 검토합니다:
- AC가 코드로 검증 가능한가?
- 모호하거나 모순되는 부분은 없는가?
- 영향 범위가 충분히 파악되었는가?
- 엣지 케이스가 고려되었는가?

결과: **PASS** / **CONCERNS** / **REWORK** / **FAIL**

### 2.3 구현 계획 (자동)

review-req 통과 후 자동 실행됩니다.

```
/forge-flow:plan
```

수행 내용:
1. 코드 분석 → 영향 범위 파악 (파일 경로 근거 제시)
2. 확정 규모 판정 (S/M/L)
3. 변경 파일 목록, 순서, 리스크 정리
4. 기능 브랜치 생성
5. 병렬 가능성 판단 → 에이전트팀 제안 (조건 충족 시)

### 2.4 설계 검수 (조건부)

```
/forge-flow:review-plan
```

실행 조건:
- **L 규모**: 항상 실행
- **M 규모**: 새 의존성, API 변경, DB 스키마 변경, 3개 이상 모듈 영향, 새 패턴 도입 시
- **S 규모**: 건너뜀

### 2.5 구현

plan 완료 후 자유롭게 구현합니다.
워크플로 상태는 `implementing`으로 추적됩니다.

구현 중 참고:
- 프롬프트마다 현재 워크플로 상태가 자동 알림됩니다
- 위험한 명령(`rm -rf`, `git push --force` 등)은 자동 차단됩니다
- 구현 완료 시 `/forge-flow:verify` 실행이 필요합니다

### 2.6 검수

```
/forge-flow:verify
```

규모별 검수 방식:
- **S**: 빌드 검증 + AC 스팟 체크
- **M**: 빌드 + 코드 리뷰 + 서브에이전트 교차검증
- **에이전트팀**: 리뷰어 + 메인 세션 + 서브에이전트 3중 검증

검수 통과(PASS) 시 커밋이 가능해집니다.
미통과 시 `implementing`으로 돌아가 수정 후 재검수합니다.

### 2.7 커밋

verify 통과 후 커밋합니다. 워크플로가 완료(`completed`) 상태가 되며 세션을 종료할 수 있습니다.

---

## 3. 품질 게이트

모든 검수 단계(review-req, review-plan, verify)는 동일한 4단계 게이트를 사용합니다:

| 판정 | 의미 | 후속 조치 |
|------|------|----------|
| **PASS** | 통과 | 다음 단계 진행 |
| **CONCERNS** | 경미한 우려 | 사용자에게 알리고 판단 위임 |
| **REWORK** | 수정 필요 | 해당 단계 재작업 → 재검수 |
| **FAIL** | 심각한 결함 | 이전 단계로 돌아감 |

REWORK이 3회 연속 발생하면 FAIL로 에스컬레이션됩니다.

---

## 4. 설계 문서

모든 작업은 `design/{작업명}.md`에 기록됩니다.

포함 내용:
- 요구사항 정리
- 수용 기준(AC)
- 영향 범위
- 제외 범위
- 리스크
- 규모 판정
- 구현 계획 (plan 이후)
- 검수 결과/이력

`design/` 디렉토리는 git 추적 대상이므로 팀과 공유할 수 있습니다.

---

## 5. 훅 동작

설치 시 3개의 훅이 자동 등록됩니다:

| 훅 | 시점 | 동작 |
|----|------|------|
| **workflow-state** | 매 프롬프트 | 현재 워크플로 상태를 컨텍스트에 주입 |
| **stop-guard** | 세션 종료 시 | 미완료 워크플로면 종료 차단 (3회 연속 시 강제 허용) |
| **dangerous-cmd-guard** | Bash 실행 전 | `rm -rf`, `DROP TABLE` 등 위험 명령 차단 |

훅 스크립트는 `jq`를 우선 사용하며, 미설치 시 `python3`으로 자동 폴백합니다.

---

## 6. 이전 세션 이어받기

새 Claude Code 세션을 열었을 때 이전 작업의 설계 문서가 남아 있으면 자동 감지됩니다:

```
[WORKFLOW] 이전 작업 감지: login-feature.md. 이어서 진행하려면 알려주세요.
```

이어서 진행하겠다고 하면 해당 설계 문서를 기반으로 워크플로를 재개합니다.

---

## 7. 규모(Scale) 판정 기준

| 규모 | 기준 | 검수 수준 |
|------|------|----------|
| **S** | 1-2 파일 변경, 기존 패턴 내, 새 의존성 없음 | 경량 (빌드 + AC 체크) |
| **M** | 3-10 파일, 로직 수정, 테스트 필요 | 표준 (코드 리뷰 + 서브에이전트) |
| **L** | 10+ 파일, 아키텍처 변경, 모듈 간 영향 | 전체 (설계 검수 + 3중 검증) |

규모는 2단계로 결정됩니다:
1. **예비 판정** (clarify): 요구사항 기반 추정
2. **확정 판정** (plan): 코드 분석 기반, 파일 경로 근거 제시

---

## 8. 에이전트팀 (선택)

install 시 에이전트팀 기능을 활성화하면, 병렬 가능한 M/L 작업에서 여러 에이전트가 동시에 구현+리뷰를 수행합니다.

활성화 조건 (모두 충족):
1. install에서 에이전트팀을 활성화함
2. S 규모가 아님
3. 2개 이상 독립 모듈로 병렬 작업 가능

기본값은 비활성화(단일 세션 워크플로)입니다.

---

## 9. 자주 묻는 질문

**Q. 워크플로 없이 빠르게 작업하고 싶다면?**
A. `/forge-flow:install --purge`로 제거하거나, 설치하지 않은 프로젝트에서는 워크플로가 동작하지 않습니다.

**Q. 검수에서 계속 REWORK이 나온다면?**
A. 3회 연속 REWORK 시 FAIL로 에스컬레이션되며, 이전 단계로 돌아갑니다. 요구사항이나 설계를 재검토하세요.

**Q. 세션이 종료되지 않는다면?**
A. stop-guard 훅이 미완료 워크플로의 종료를 차단합니다. `/forge-flow:verify`를 통과하거나, 3회 연속 종료 시도 시 강제 허용됩니다.

**Q. 플러그인을 업데이트했는데 프로젝트에 반영이 안 된다면?**
A. 스킬 정의(SKILL.md)는 즉시 반영됩니다. 훅 스크립트는 프로젝트에 복사된 것이므로 `/forge-flow:install --update`로 갱신하세요.
